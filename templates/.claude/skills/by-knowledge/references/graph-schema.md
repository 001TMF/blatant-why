# Graph Schema Reference

Full JSON Schema for every entity and relationship type in the BY knowledge
graph, plus field-level documentation and a property index for fast lookup.

The on-disk format is two JSON arrays (`campaigns.json`, `failures.json`) under
`~/.by/knowledge/` (or override path). Conceptually, the graph contains five
entity types and five relationship types, with campaigns acting as the central
hub. The migration script (`scripts/migrate_knowledge.py`) normalizes between
the on-disk arrays and the conceptual schema below.

---

## Schema Versioning

Current schema version: `2.0`

| Version | Date | Changes |
|---------|------|---------|
| `1.0` | 2025-Q4 | Initial: campaigns + failures, no designs array |
| `2.0` | 2026-Q2 | Added top-level `designs` array on Campaign; added `composite_score` |

The migration script handles `1.0 -> 2.0` automatically: missing `designs`
becomes `[]`, missing `composite_score` is computed from `0.50*ipsae +
0.30*iptm + 0.20*(1 - liabilities/10)`.

## Entity Types

### Campaign

Top-level record of a completed (or partially completed) design campaign.
Persisted to `campaigns.json`.

```json
{
  "$id": "campaign",
  "type": "object",
  "required": ["id", "target", "modality", "parameters", "outcomes", "stored_at"],
  "properties": {
    "id": {
      "type": "string",
      "pattern": "^campaign_[a-f0-9]{12}$",
      "description": "Auto-generated by knowledge_store_campaign"
    },
    "target": {
      "type": "string",
      "description": "Lowercase-hyphenated target name (e.g. 'tnf-alpha')"
    },
    "modality": {
      "type": "string",
      "enum": ["antibody", "nanobody", "VHH", "scFv", "de_novo", "binder"]
    },
    "parameters": {
      "type": "object",
      "properties": {
        "scaffold": {"type": "string"},
        "seeds": {"type": "integer", "minimum": 1},
        "temperature": {"type": "number", "minimum": 0.0, "maximum": 2.0},
        "msa_mode": {"type": "string", "enum": ["paired", "unpaired", "none"]},
        "tier": {"type": "string", "enum": ["preview", "standard", "production"]},
        "num_designs": {"type": "integer", "minimum": 1}
      },
      "additionalProperties": true
    },
    "outcomes": {
      "type": "object",
      "properties": {
        "hit_rate": {"type": "number", "minimum": 0.0, "maximum": 1.0},
        "best_ipsae": {"type": "number", "minimum": 0.0, "maximum": 1.0},
        "best_iptm": {"type": "number", "minimum": 0.0, "maximum": 1.0},
        "screening_pass_rate": {"type": "number", "minimum": 0.0, "maximum": 1.0}
      }
    },
    "notes": {"type": "string"},
    "designs": {
      "type": "array",
      "items": {"$ref": "#/$defs/design"},
      "description": "Top 10-20 designs only. Do not store all of them."
    },
    "stored_at": {
      "type": "number",
      "description": "Unix epoch seconds when first stored"
    },
    "access_count": {
      "type": "integer",
      "minimum": 0,
      "description": "Incremented on every knowledge_query_similar match. Drives pruning."
    }
  }
}
```

**Field guidance:**
- `target`: ALWAYS use the canonical lowercase-hyphenated form. Inconsistency here is the #1 source of fragmented data.
- `parameters.scaffold`: Required for `knowledge_scaffold_rankings` to return useful output.
- `outcomes.hit_rate`: Fraction of designs that pass screening. `screening_pass_rate` is the same metric named for legacy consumers.
- `notes`: Free-text. Indexed for keyword search.
- `access_count`: Auto-managed. Do not write directly. Used by `knowledge_consolidate` to identify pruning candidates (`>90 days AND <3 accesses`).

### Design

A single design within a campaign's top-K. Embedded in the parent campaign's
`designs` array, not stored separately.

```json
{
  "$id": "design",
  "type": "object",
  "required": ["design_id", "scaffold", "status"],
  "properties": {
    "design_id": {
      "type": "string",
      "description": "Stable identifier within the campaign (e.g. 'tnf_001')"
    },
    "job_id": {"type": "string", "description": "Compute job ID for traceback"},
    "scaffold": {"type": "string"},
    "epitope": {
      "type": "string",
      "description": "Range notation (e.g. 'A56-A58,A113')"
    },
    "tool": {
      "type": "string",
      "enum": ["BoltzGen", "PXDesign", "Protenix"]
    },
    "ipsae": {"type": "number", "minimum": 0.0, "maximum": 1.0},
    "iptm": {"type": "number", "minimum": 0.0, "maximum": 1.0},
    "plddt": {"type": "number", "minimum": 0.0, "maximum": 100.0},
    "rmsd": {"type": "number", "minimum": 0.0},
    "composite_score": {
      "type": "number",
      "minimum": 0.0,
      "maximum": 1.0,
      "description": "0.50*ipsae + 0.30*iptm + 0.20*(1 - normalized_liability_count)"
    },
    "liabilities": {
      "type": "array",
      "items": {"type": "string"}
    },
    "status": {"type": "string", "enum": ["PASS", "FAIL"]}
  }
}
```

**Selectivity rule:** Top 10-20 designs by `composite_score` per campaign.
Designs with notable liabilities or downstream usage may break this rule.

### Failure

A recurring failure pattern from a campaign. Persisted to `failures.json`.

```json
{
  "$id": "failure",
  "type": "object",
  "required": ["id", "campaign_id", "description", "root_cause", "target", "stored_at"],
  "properties": {
    "id": {
      "type": "string",
      "pattern": "^failure_[a-f0-9]{12}$"
    },
    "campaign_id": {
      "type": "string",
      "description": "Stable identifier of the source campaign (typically campaign dir name)"
    },
    "description": {"type": "string", "description": "What went wrong (1-2 sentences)"},
    "root_cause": {"type": "string", "description": "Underlying cause from post-mortem"},
    "target": {"type": "string"},
    "stored_at": {"type": "number"},
    "access_count": {"type": "integer", "minimum": 0}
  }
}
```

**Examples of useful failures (record):**
- "VHH+caplacizumab on glycosylated epitopes consistently misfolds"
- "PXDesign de novo binders on shallow grooves fail ipSAE > 0.6"
- "Multi-seed Protenix predictions diverge on flexible loops"

**Examples of non-failures (don't record):**
- "Ran out of disk on day 3" (infra issue)
- "User passed wrong PDB chain" (user error)
- "Pip install of jsonschema failed" (env issue)

### Target (conceptual)

Not stored as a standalone entity in v2.0 — derived from `Campaign.target`.
Reserved for v3.0 when target-level metadata (organism, difficulty, UniProt) is
promoted out of campaign records.

```json
{
  "$id": "target",
  "type": "object",
  "properties": {
    "name": {"type": "string"},
    "uniprot_accession": {"type": "string", "pattern": "^[OPQ][0-9][A-Z0-9]{3}[0-9]|[A-NR-Z][0-9]([A-Z][A-Z0-9]{2}[0-9]){1,2}$"},
    "pdb_ids": {"type": "array", "items": {"type": "string", "pattern": "^[0-9A-Z]{4}$"}},
    "organism": {"type": "string"},
    "target_type": {
      "type": "string",
      "enum": ["cytokine", "receptor", "enzyme", "structural", "viral", "toxin", "other"]
    },
    "difficulty": {
      "type": "string",
      "enum": ["well-studied", "moderate", "novel", "ultra-novel"]
    }
  }
}
```

When promoting to v3.0, the migration script will extract target records from
existing campaigns by grouping on `target` name.

### Scaffold (conceptual)

Not stored as a standalone entity in v2.0 — derived from
`Campaign.parameters.scaffold`. Aggregated stats come from
`knowledge_scaffold_rankings`.

```json
{
  "$id": "scaffold",
  "type": "object",
  "properties": {
    "name": {"type": "string"},
    "modality": {"type": "string", "enum": ["VHH", "scFv", "Fab", "de_novo"]},
    "source": {"type": "string", "description": "Origin (e.g. 'caplacizumab', 'ozoralizumab')"},
    "framework": {"type": "string", "description": "Germline family if antibody"}
  }
}
```

## Relationship Types

In v2.0, relationships are implicit: a campaign record contains its scaffold,
designs, and target. The conceptual graph nonetheless has these edges, exposed
via query patterns rather than stored as separate edge records.

| Edge | From -> To | Implicit Location | Query Tool |
|------|-----------|-------------------|------------|
| `targets_epitope` | Target -> Epitope | Campaign.parameters (hotspot range notation) | Manual; not in v2.0 query API |
| `uses_scaffold` | Target -> Scaffold | Campaign.parameters.scaffold + Campaign.target | `knowledge_scaffold_rankings(target_class)` |
| `produces_design` | Scaffold -> Design | Campaign.designs[].scaffold == Campaign.parameters.scaffold | `knowledge_query_similar` then drill into `metadata.designs_count` |
| `has_result` | Design -> ScreenResult | Embedded: Design.ipsae, Design.iptm, Design.status | Iterate `Campaign.designs[]` |
| `exhibits_failure` | Design/Scaffold -> FailurePattern | Failure.campaign_id links back to Campaign | `knowledge_get_recommendations` warnings array |

For v3.0 promotion, see `scripts/migrate_knowledge.py --upgrade-relationships`.

## On-Disk Format

Two JSON arrays:

```
~/.by/knowledge/
  campaigns.json   # JSON array of Campaign records
  failures.json    # JSON array of Failure records
```

**Writes** are atomic: server writes to `.tmp`, then renames. Concurrent writes
are safe at the rename boundary; reads during a write see the pre-write state.

**Migration to NDJSON** (one entity per line) is supported via
`migrate_knowledge.py --export` for external analytics tools. The migration
script also handles the reverse — ingesting NDJSON dumps back into the JSON-array
format.

---

## Field-Level Validation Rules

Enforced by `migrate_knowledge.py --validate`:

1. `target` MUST be lowercase, with hyphens (not spaces or underscores).
2. `modality` MUST be one of the enum values.
3. `outcomes.*` numeric fields MUST be in `[0.0, 1.0]`.
4. `designs[]` MUST have <= 20 entries (warns at 15, errors at 25).
5. `Design.composite_score` MUST be present in v2.0+ (auto-computed if missing).
6. `Failure.description` and `Failure.root_cause` MUST be non-empty.
7. `id` MUST match the `^(campaign|failure)_[a-f0-9]{12}$` pattern.
8. `stored_at` MUST be a unix epoch float > 1.6e9 (post-2020 sanity check).

Violations are logged as warnings; the migration script can be run with
`--strict` to fail on any violation.
